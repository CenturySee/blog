# Agent进化论：从Tools到Skills，Claude在下什么棋？

大家好，我是算法工程笔记。

最近 Claude 推出的 Skills 功能刷屏了技术圈。很多朋友问我：这玩意儿不就是 Tool Use 吗？换个马甲我就不认识了？

其实不然。如果说 Tools 是 Agent 的"手"，那么 Skills 更像是让 Agent 拥有了**"外挂大脑"**。今天这篇主要聊聊，为什么 Skills 是 Agent 走向复杂工程化的关键一步。

## 1. 什么是 Skills？

简单来说，Skills 是一种**可动态加载**的"能力插件"或"知识包"。

形式上，它就是一个文件夹，里面整整齐齐地放着：
*   **指令 (Instructions)**: 比如 `SKILL.md`，写满了私有化的 SOP（标准作业程序）。
*   **脚本 (Scripts)**: Python 脚本等，用来执行确定性的操作。
*   **资源 (Resources)**: 其他辅助文件。

Agent 就像一个"乐高小人"，需要什么能力，就"咔嚓"一声把对应的 Skills 模块插上去。

## 2. 为什么我们需要 Skills？

在 Agent 开发中，我们苦"Context 爆炸"久矣。Skills 的出现，主要是为了解决从 Demo 到 Production 的两个核心痛点。

### 1. 渐进式披露：给大脑"减负"

**没有 Skills 前**（单体应用模式）：
我们需要把所有的业务逻辑、具体的 Prompt、几十个 Tool 的定义，全部一股脑塞进 System Prompt 里。这就好比让一个员工在入职第一天，就把公司几百页的规章制度倒背如流。结果就是：
*   **贵**：Token 消耗巨大。
*   **笨**：上下文太长，模型注意力分散，极易产生幻觉。

**有了 Skills 后**（微服务模式）：
Agent 采用了 **"渐进式披露" (Progressive Disclosure)** 的策略。
*   **Level 1**: 启动时，Agent 只知道兜里揣着个锦囊叫"PDF处理技能"，不知道具体怎么用。
*   **Level 2**: 只有当真的需要处理 PDF 时，它才会打开锦囊，动态加载里面的详细说明书和代码。

这种机制彻底打破了 Context Window 的瓶颈，**理论上你可以给 Agent 挂载成百上千个 Skills，而平常它依然保持"轻装上阵"。**

![skill渐进式披露的三个阶段.webp](https://image.imak.top/2026/01/01bada280163aeaa1b529e8cdfedf8ea.webp)

### 2. 标准化复用：拒绝"重复造轮子"

**没有 Skills 前**：
你想做一个"合同审核 Agent"，辛辛苦苦调好了 Prompt，挂好了 Tools。下个月要做"发票处理 Agent"，又得从头来一遍。能力分散且难以迁移。

**有了 Skills 后**：
你把"合同审核"这一套 **SOP + 脚本** 打包成一个 Skill 文件夹。
这就变成了标准的交付物。任何通用的 Claude Agent，只要挂载这个文件夹，瞬间就拥有了这项专家能力。**一次编写，到处挂载**，就像我们管理 Python 库一样管理 AI 能力。

### 3. 确定性与灵活性的结合

Skills 允许你把 **确定性的代码** (Scripts) 和 **非确定性的推理** (Prompt) 强绑定。
对于计算、I/O 这类任务，Skill 可以强制 Agent 走代码逻辑，而不是靠 LLM "脑补"，从而大幅提升了复杂任务的成功率。

## 3. Skills vs. Tools：到底强在哪？

很多同学容易混淆，这里做一个直观的对比。

> **Tools 是原子化的"动作"，而 Skills 是封装了"动作 + 策略 + 知识"的完整"模块"。**

如果用编程来打比方：
*   **Tools 就像 API 函数** (如 `requests.get`)。它只是一个执行动作。
*   **Skills 就像封装好的 Library**。它不仅包含了底层的 API，还包含了文档说明、最佳实践（SOP）、以及与之配合的代码逻辑。

Skills 并没有取代 Tools，而是**管理和组织 Tools 的一种更高层级的架构**。
